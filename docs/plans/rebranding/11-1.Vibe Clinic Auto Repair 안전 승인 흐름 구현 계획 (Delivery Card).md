# Vibe Clinic Auto Repair 안전 승인 흐름 구현 계획 (Delivery Card)

이 문서는 사용자의 "사전 승인 후 코드 적용" 피드백을 반영하여, AI Auto Repair 기능을 `propose -> review -> approve/apply` 3단계 안전 워크플로우로 개편하기 위한 Decision Memo 및 구현 계획서입니다.

---

## 1. Decision Memo (의사결정 검토)

### 비교 분석 및 기술 선택

- **Diff 렌더링 방식**: 외부 무거운 라이브러리(Monaco Editor, diff-match-patch 등) 대신, 기존에 작성되어 정상 동작 중인 **순수 CSS/JS 인라인 디프 렌더러(computeSimpleDiff)를 확장(Option B)**하여 사용합니다. 이는 대시보드를 가볍고 자립적(Self-contained)으로 유지하기 위함이며, 향후 대규모 코드 이동이나 특이 케이스에서 가독성 한계가 드러날 경우에만 외부 라이브러리 도입을 재검토합니다.
- **제안(Proposal) 상태 관리**: 제안 정보는 서버 메모리 상에 `Map` 구조로 관리합니다. 파일에 기록하지 않으므로 디스크 오염이 방지됩니다.
- **안전 장치**:
  1. **원본 해시(Hash) 검증**: 제안 시점의 원본 파일 내용 해시와 승인 시점의 현재 파일 내용 해시를 비교하여 다를 경우(사용자가 그사이 파일을 수정함) 승인을 거부합니다.
  2. **시간 만료**: 제안 생성 후 10분(600,000ms)이 초과하면 승인 요청 시 거부하며 메모리에서 삭제합니다.
  3. **VS Code 직접 수정 제거**: VS Code에서 자동 수리 명령 수행 시, 파일을 직접 수정하는 API 호출을 우회 처리하여 대시보드 URL(`?repair=diagId`)로 사용자를 안내하여 단일 승인 창구를 유지합니다.

### 의사결정 게이트 (Decision Gate)
- **상태**: `Go`
- **결정 소유자**: 윤겸스
- **근거**: AI 자동 수리의 위험성(코드 덮어쓰기)을 완벽하게 예방하고 사용자의 통제권을 복원하기 위한 필수적 조치입니다.

## 2. Delivery Card (실행 명세)

### 작업 범위 (Scope)
- 서버 측 수리 로직 분리: 제안 생성(propose), 승인 적용(approve), 취소(cancel) 기능 구현.
- 대시보드 REST API 신설 및 연동.
- 대시보드 웹 UI에 승인(Approve & Apply) / 취소(Cancel) 버튼 배치 및 상태 제어.
- VS Code 확장의 Auto Repair 명령어를 대시보드(`http://localhost:7700/?repair=diagId`) 자동 팝업 흐름으로 리다이렉트.
- 사전 승인 흐름의 무결성을 검증하기 위한 자동 테스트 추가.

### 수락 기준 (Acceptance Criteria)
1. **제안/취소 시 0건 변경**: `/api/repair/propose` 호출 후 또는 `/api/repair/cancel` 호출 완료 후, 대상 프로젝트 파일 및 `.bak` 파일의 생성/변경이 전혀 없어야 합니다.
2. **승인 시에만 백업 및 덮어쓰기**: `/api/repair/approve` 호출 시에만 디바이스에 `.bak` 파일이 생성되고, 수정된 내용이 본 파일에 적용되며, 진단이 재실행되어야 합니다.
3. **만료 제안 적용 거부**: 제안 생성 후 10분이 지난 상태에서 승인을 요청하면 거부되고 파일이 수정되지 않아야 합니다.
4. **원본 변경 시 승인 거부**: 제안 생성 후 파일이 외부(수동 편집 등)에서 수정되어 해시가 변경된 경우, 승인이 거부되고 AI 수정안이 반영되지 않아야 합니다.
5. **VS Code 연동성**: VS Code 확장프로그램에서 `Auto Repair` 실행 시 QuickPick으로 에러를 고르면 대시보드 브라우저 창이 열리며 즉시 해당 에러 복구 제안 모달이 화면에 표시되어야 합니다.

## 4. Verification Plan (검증 및 테스트 계획)

### Automated Tests
`test/repair-flow.test.js` 파일을 새로 생성하여 아래의 비즈니스 시나리오를 코드로 철저히 검증합니다:
1. `proposeRepair` API 실행 후 원본 파일이 전혀 변경되지 않고 백업 파일도 생성되지 않음을 검증.
2. `applyRepair` 성공 시 파일이 AI가 제안한 내용으로 변경되고 `.bak` 백업 파일이 올바르게 생성됨을 검증.
3. 수리 대상 파일의 내용이 제안 시점과 달라진 경우, `applyRepair`가 에러를 뱉고 적용되지 않음을 검증.
4. 제안 시점으로부터 10분이 경과한 경우, `applyRepair`가 만료 에러를 뱉고 적용되지 않음을 검증.
5. `cancelRepair` 호출 시 메모리에서 제안이 삭제되고 파일은 변함이 없음을 검증.

### Manual Verification
1. 계산기 예제(`examples/calculator`) 중 일부를 강제로 실패하도록 수정합니다.
2. 대시보드를 실행하고 실패한 테스트 카드의 🔧 자동 복구 버튼을 클릭합니다.
3. 코드 디프 뷰어 모달이 열리면 제안된 변경 사항을 확인하고 **[취소]**를 눌러봅니다. 파일 변경이 없는지 에디터에서 확인합니다.
4. 다시 수리를 시도하여 디프 뷰어 모달을 띄운 뒤, 이번에는 **[수락 및 적용]**을 클릭합니다. 파일이 정상적으로 수정되고, `.bak` 백업이 생성되며, 대시보드의 상태가 녹색(OK)으로 갱신되는지 확인합니다.
5. VS Code에서 `Vibe Clinic: Auto Repair` 명령을 호출하여 QuickPick 선택 시 대시보드 브라우저 창이 열리며 즉시 수리 제안 모달이 뜨는지 점검합니다.
