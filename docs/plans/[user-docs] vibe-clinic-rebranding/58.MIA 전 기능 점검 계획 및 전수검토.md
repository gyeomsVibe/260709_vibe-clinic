# MIA 전 기능 점검 계획 및 전수검토 (Full Feature Audit)

> Stage: 검토(檢討) · Status: `IN_PROGRESS`
> 기준일: 2026-07-17 · 결정권자: 윤겸스
> 계기: "v1, v2 모양만 그럴듯하고 모든 기능이 작동 안한다" + 폴더 선택기가 브라우저 뒤에 뜸

## 0. 점검 원칙

- **실측만 인정한다.** 단위 테스트 통과 ≠ 기능 작동. 반드시 실행 중인 서버 + 실제 UI 클릭으로 확인한다.
- **1 실패 → 1 원인 → 최소 수정 → 재검증.**
- 테스트를 약화시켜 통과시키지 않는다.
- 판정: ✅작동 / ❌고장 / ⚠️조건부(제약 있음) / ⛔미검증

## 2. 확정된 결함

### ❌ C4 — 폴더 선택기가 브라우저 뒤에 뜬다 (실사용 불가)

- **증상**: [선택] 클릭 → 대화창이 브라우저/에디터 **뒤에** 생성. 사용자는 "안 열린다"고 인식.
- **근본 원인**: [backend/src/folder-picker.ps1](../../backend/src/folder-picker.ps1) 67–69행
  ```powershell
  $owner = New-Object System.Windows.Forms.Form
  $owner.ShowInTaskbar = $false
  $ownerHandle = $owner.Handle   # 핸들만 만들고 Show()/Activate() 안 함
  ```
  owner 창이 **표시·활성화되지 않아** Windows 포그라운드 정책상 소유 대화창이 앞으로 나올 수 없다.
- **방치 이력**: 같은 파일 63–66행 주석이 *"포그라운드 정책상 브라우저 뒤에 놓일 수 있다… 보조 수단으로 취급"*이라고 **인지하고도 미해결**. UI는 *"보이지 않으면 작업 표시줄을 확인하세요"* 토스트로 사용자에게 전가.
- **수정 방침**: owner를 **화면 밖(-32000,-32000) + Opacity 0 + TopMost**로 실제 `Show()`·`Activate()` → 잔상(ghost) 없이 포그라운드 자격 획득 → 대화창이 최상단에 표시. 과거 "잔상" 문제의 원인이던 *보이는 창*은 만들지 않는다.
- **수용 기준**: 대화창이 **다른 창을 가리고 최상단**에 뜬다 / 선택·취소 정상 / 잔상 없음 / `-DryRun` = `DRYRUN_OK` 유지.

## 3. C4 수정 이력 (실패 2회 → 교정)

| 시도 | 방법 | 결과 |
|---|---|---|
| 1차 | owner를 화면 밖·Opacity0·TopMost로 `Show()` + `AttachThreadInput`+`SetForegroundWindow` | ❌ 실패. 윤겸스 확인 "v1·v2 둘 다 브라우저 뒤". **오판**: Node가 띄운 자식 프로세스는 포그라운드 권한 자체가 없어 owner를 어떻게 다뤄도 안 됨 |
| 2차 | 워커 러너스페이스가 **창 제목**으로 `FindWindow` → `HWND_TOPMOST` | ⚠️ 제목 매칭 불안정(샌드박스에서 창 탐색 실패). 미검증 |
| 3차 | Add-Type P/Invoke + 워커 러너스페이스가 `GetWindow(GW_ENABLEDPOPUP)`→`HWND_TOPMOST` | ❌ **실패 + 2차 피해**. 여전히 뒤에 뜸 → 사용자는 창을 못 봄 → 5분 타임아웃이 프로세스 강제종료 → `/api/project/select` **500**. 파일의 설계 제약("compile-free, AV-safe")도 위반 |
| **4차(현행)** | **HEAD(검증된 원본) 복원 + 한 줄**: owner 핸들 생성 **전** `$owner.TopMost = $true` → `WS_EX_TOPMOST`가 창 생성에 포함 → 소유 대화창이 owner의 topmost 밴드 상속. Add-Type·러너스페이스 전부 제거(compile-free 제약 준수) | ✅ DryRun OK · 서버 동일 경로(Node execFile)에서 크래시 없이 대화창 대기 확인 · 테스트 46/46. **육안 검증 대기** |

**핵심 원리**: `WS_EX_TOPMOST`(항상 위)는 **포그라운드 권한이 필요 없다** — 백그라운드 서버의 자식 프로세스도 쓸 수 있는 유일한 z-order 수단. `SetForegroundWindow` 계열은 전부 정책상 거부됨.
**500의 정체**: 대화창이 뒤에 떠서 안 보임 → 사용자는 "안 열림"으로 인식 → 5분 타임아웃 → Node가 프로세스 kill → `err.killed` → 500. **z-order 문제와 500은 같은 뿌리.**

### ⚠️ 검증 한계 (정직 고지)
에이전트 샌드박스 셸에서는 **GUI 창을 관측할 수 없다**. 대화창이 열려 블록되는 것(exit 124)은 확인했으나, **"브라우저 앞에 뜨는지"는 윤겸스의 육안 확인이 유일한 검증 수단**이다. 자동 테스트로 대체 불가.

## 3-1. C4 최종 해결 — 아키텍처 교체 (2026-07-18, 윤겸스 A안 승인)

4차(TopMost owner)와 병렬 세션의 5번째 변형까지 **전부 실패**함으로써, OS 대화창 방식은 구현 버그가 아니라 **아키텍처 벽**으로 확정: *백그라운드 서버의 자식 프로세스는 Windows 포그라운드 정책상 대화창을 브라우저 앞에 안정적으로 띄울 수 없다.*

**해결**: OS 대화창 **완전 삭제** → **웹 내장 폴더 탐색기**로 재건축.
- 삭제: `folder-picker.ps1`(122줄), `runFolderPicker`/`parseFolderPickerOutput`, 관련 테스트 3건. `POST /api/project/select` → **410** + 안내(레거시 V1 대응).
- 신설: `GET /api/fs/list`(읽기 전용: 드라이브 루트/하위 폴더, 400 검증, 접근불가 폴더 제외) + V2 폴더 탐색 모달(브레드크럼·상위/하위 탐색·[이 폴더 선택] → 기존 `changeProject` 재사용).
- 검증 (전부 자동, 육안 의존 소멸): 백엔드 45/45 + 계약 테스트(fs/list 3종·select 410) / 린트 0 / 빌드 성공 / **브라우저 E2E 완주** — [폴더 탐색] 클릭 → 모달이 **대시보드 안에** 표시 → 상위·하위 탐색 → 선택 확정 → 모달 닫힘 → 프로젝트 변경 → 자동 진단 연쇄(진단 전 3 → 정상 3).
- **z-order 문제는 원리적으로 소멸** (창이 브라우저 밖으로 나가지 않음).
