## 핵심 차이

### 1. VS Code 확장의 CLI 실행

| 관점 | Vibe Diagnosis 원작 | Vibe Clinic 2.0 |
|---|---|---|
| 실행 API | `child_process.exec` | `child_process.execFile` |
| 인자 전달 | 셸 명령 문자열 | 실행 파일과 인자 배열 |
| CLI 탐색 실패 | `npx vibe-diag` 폴백 | 오류를 명시하고 종료 |
| 사용자 경로 설정 | 확인 스냅샷에 없음 | `vibeClinic.cliPath` |

Vibe Clinic 방식은 셸 인용과 경로 공백에 대한 의존을 줄인다. CLI가 없을 때 네트워크나 npm 캐시에 기대지 않고 필요한 설정을 사용자에게 알리는 것도 현재 저장소의 로컬 우선 운영 정책과 맞는다.

### 2. 대시보드 수명 주기

| 관점 | Vibe Diagnosis 원작 | Vibe Clinic 2.0 |
|---|---|---|
| 실행 방식 | `exec`, 5초 타임아웃 | `spawn` |
| 프로세스 분리 | 명시적 분리 없음 | `detached: true`, `unref()` |
| 표준 입출력 | 콜백 기반 | `stdio: 'ignore'` |

Vibe Clinic은 장기 실행 서버를 확장 명령의 단기 실행과 분리한다. 다만 OS별 종료·포트 회수 동작은 VS Code GUI 통합 테스트로 추가 확인할 필요가 있다.

### 3. 명칭과 상태 경계

Vibe Clinic은 다음 공개 인터페이스를 `vibe-clinic` 기준으로 통일했다.

- CLI: `vibe-clinic`, 별칭 `vbc`
- 상태 디렉터리: `.vibe-clinic/`
- 진단 파일: `*.clinic.js`
- MCP 도구: `init_clinic`, `list_clinics`, `run_clinic`
- VS Code 명령: `vibeClinic.*`
- 환경 변수: `VIBE_CLINIC_*`

기존 `.vibe-diagnosis/`와 `*.diag.js`는 읽지 않는다. 따라서 같은 프로젝트에서 이전 상태를 자동 승계하려면 사용자가 직접 변환해야 한다.

### 4. 승인 기반 에이전트 규칙

Vibe Clinic은 Claude Code와 Antigravity/Gemini가 같은 원터치 점검 절차를 따르도록 프로젝트 규칙 파일을 제공한다.

- 기준 문서: `GEMINI.md`
- 프로젝트 스킬: `.claude/skills/vibe-check/SKILL.md`
- 로컬 동기화: `npm run sync:rules`
- 사용자 전역 동기화: `npm run sync:rules:global`

기본 명령은 프로젝트 내부만 수정한다. 사용자 홈의 전역 스킬 변경은 명시적인 `:global` 명령으로 분리해 승인 경계를 분명히 한다.

## 현재 검증 결과

Vibe Clinic `2.0.0` 작업 트리에서 다음 항목을 검증 대상으로 삼는다.

- 단위 테스트: `npm test`
- 저장소 자체 진단: `node bin/vibe-clinic.js run`
- 예제 프로젝트 진단: `node bin/vibe-clinic.js run --cwd examples/calculator`
- MCP 스모크 테스트: `node mcp-server/test-mcp.cjs`
- VSIX 패키지 메타데이터와 포함 파일 검사

VS Code GUI에 VSIX를 직접 설치하고 명령을 클릭하는 수동 통합 테스트는 별도 항목이다. npm 공개 배포, 마켓플레이스 배포, 릴리스 생성도 이 보고서의 검증 범위에 포함하지 않는다.

## 결론

윤겸스의 목표가 로컬 소스 기반 운영, Vibe Clinic 전용 명칭, 승인 기반 자동 점검, 셸 의존 축소라면 현재 `2.0.0` 분기가 더 적합하다. 원작은 프로젝트의 출발점이자 비교 기준으로 계속 크레딧에 보존한다. 어느 구현이 보편적으로 더 낫다고 단정하기보다는, 현재 운영 요구와 검증된 차이를 기준으로 Vibe Clinic을 선택한다.
