## 2. 핵심 결함 진단 및 처방

### ❌ [C4] 폴더 선택기가 브라우저/에디터 뒤에 뜨는 문제 (실사용 불가)

#### 1) 증상 및 원인 분석
- **증상**: 대시보드 UI에서 `[선택]`을 누르면 폴더 선택창이 뒤쪽에 생성되어 보이지 않음. 사용자는 멈춘 것으로 오인하고 결국 5분 타임아웃으로 Node 프로세스가 강제 종료되면서 API가 **500 에러**를 뿜음.
- **원인**: 백엔드에서 폴더 선택을 위해 실행하는 PowerShell 스크립트 `backend/src/folder-picker.ps1`에서 Form 객체의 Handle을 생성할 때, 윈도우 포그라운드 정책으로 인해 소유자(owner)가 활성화되지 않아 자식 창(FolderBrowserDialog)이 맨 앞으로 오지 못함.
- **기존 실패 이력**:
  - *1차 시도*: `AttachThreadInput` / `SetForegroundWindow` 적용 -> 실패. (백그라운드 Node 프로세스로 실행되므로 권한이 없어 무시됨)
  - *2차 시도*: 창 제목으로 `FindWindow` -> `HWND_TOPMOST` 설정 -> 실패. (샌드박스 환경 내 창 매칭 불안정)
  - *3차 시도*: csc.exe를 통한 P/Invoke DLL 컴파일 및 `GetWindow` -> 실패. (500 에러 및 컴파일 프리 제약 위반)

#### 2) 최종 처방 (4차 수정 - 적용 상태)
- **해결 방안**: owner Form 핸들이 생성되기 **직전**에 `$owner.TopMost = $true`를 설정합니다.
- **원리**: `WS_EX_TOPMOST` 스타일이 창 생성 초기 속성에 포함되면, 자식 대화창이 owner의 topmost 밴드를 상속받아 Windows 포그라운드 획득 정책과 무관하게 항상 최상단에 뜨게 됩니다.
- **현 상태**: `backend/src/folder-picker.ps1`에 `$owner.TopMost = $true`가 반영되어 있으며, 에뮬레이터 환경에서 프로세스 대기 상태(DryRun OK)를 확인했습니다.
