## 🧭 1. Opportunity Brief

**목표**: 진단·치료 엔진(백엔드)과 대시보드(프론트엔드)를 분리해, 서로 다른 AI 도구가 **충돌 없이 병렬 작업**하도록 최적화.

**현재 구조 (실측)**:

| 진영 | 구성 | 빌드 | 실행/검증 |
|---|---|---|---|
| **백엔드** (Node, 무빌드) | `src/*.js`(runner·repairer·triage·schema·config·ai-provider·init·reporter·**dashboard 서버**), `bin/`, `mcp-server/`, `test/` | 없음 | `npm test`(41), CLI, MCP |
| **프론트엔드** (React+Vite) | `dashboard-ui/`(V1), `dashboard-ui-v2/`(V2, worktree) | Vite → `src/dist` | `npm run build` |
| **이음새(seam)** | REST 15개 엔드포인트 (`/api/*`) + dashboard.js가 `src/dist` 정적 서빙 | — | — |

**핵심 결합 문제 3가지**:
1. **API 계약이 암묵적** — 단일 진실원(SoT) 문서 없음. 프론트가 엔드포인트 형태를 추측(dashboardApi.js), 한쪽 변경이 다른 쪽을 조용히 깸.
2. **dashboard.js가 백엔드 로직 + 정적 서빙을 겸함** — 두 진영이 같은 파일을 만짐.
3. **프론트 빌드 산출물이 `src/dist`(백엔드 영역)에 떨어짐** — 경계를 넘는 쓰기 + 독립 실행 불가(프론트가 백엔드 없이 못 돎).

## ⚖️ 2. Decision Memo — 3개 대안

| 대안 | 내용 | 가치 | 실행성 | 위험 |
|---|---|---|---|---|
| **A. 즉시 물리 분리** | `src→backend/`, `dashboard-ui→frontend/` 지금 이동 | 경계 명확 | **낮음** | import 경로·CLI bin·**MCP 절대경로(~/.claude·codex·gemini)**·dist 서빙·examples 전부 파손. **Codex 정리와 충돌** |
| **B. 논리 분리 (계약+독립화)** | 폴더 유지 + API 계약 SoT + 진영별 스크립트 + vite 프록시 + 소유권 규칙 | **높음(병렬성 즉시 확보)** | **높음** | 폴더명이 문자 그대로 backend/frontend는 아님 |
| **C. 하이브리드(권장)** | **B를 먼저**(지금·저위험) → V2 병합·Codex 완료 후 **물리 분리를 원자적 git mv 한 커밋으로** | 높음+명확 | 높음 | 2단계라 일정 세분화 |

### 결정: **Go — 대안 C (B 선행 → 물리 분리 후행)**

**이유**: 병렬 작업 능력은 **계약+독립 실행**에서 나오지 폴더 위치에서 나오지 않습니다. B는 지금 당장 위험 0으로 병렬성을 열고, 물리 분리(사용자가 원하는 최종 형태)는 위험 요인(MCP 절대경로 등)이 정리된 뒤 안전하게 얹습니다.
