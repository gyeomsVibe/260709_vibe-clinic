## 1. Opportunity Brief

### 문제
현재 치료는 **개별 진단 드로어를 열고 "AI 치료 처방 요청"을 눌러 한 건씩** 처리해야 한다. 전역 치료 진입점(버튼)이 없어, 실패가 여러 건이면 사용자가 반복 클릭해야 하고 "치료가 됐다"는 확신을 얻기 어렵다.

### 원하는 결과
- 진단 실행 🩺 **바로 옆에 💉 전체 치료 버튼** 1개.
- 클릭 한 번으로 실패(ERROR/WARNING) 진단 전체를 순차 치료.
- **완치 판정은 오직 재진단 결과(status OK) 기준** — AI/로컬 처방이 "고쳤다"고 주장해도 재진단이 OK가 아니면 치료로 인정하지 않고 자동 롤백한다.

### 핵심 가설
전체 치료는 **새 치료 로직이 아니라, 이미 구축된 안전 치료 파이프라인(P2 약화 차단 + P3 회귀 게이트/자동 롤백 + VERIFIED_RESULT)을 여러 건에 순차 적용하는 오케스트레이터**다. 따라서 "실제 치료" 보장은 이미 코드로 존재하며, 이 계획은 그 위에 진입점과 배치 흐름만 얹는다.

## 2. "실제 치료"의 정의와 보장 (할루시네이션 방지) — 문서의 핵심

| 위험 | 이미 있는 방어 | 전체 치료에서의 적용 |
|---|---|---|
| AI가 "고쳤다"고만 하고 실제론 안 고침 | 성공 판정 = `applyRepairProposal`의 `maturity==='VERIFIED_RESULT'` (적용 후 재진단 status==='OK') | 각 건마다 재진단 통과분만 "완치"로 카운트 |
| 치료가 다른 진단을 파손(회귀) | P3 회귀 게이트 → 자동 롤백(`ROLLED_BACK`) | 회귀 발생 건은 원상복구 + "롤백"으로 분류, 완치 아님 |
| 진단 기대값을 완화한 "가짜 OK" | P2 `BLOCKED_WEAKENING` 가드 | 차단 건은 "약화 차단"으로 분류 |
| 파일로 못 고치는 병을 억지로 코드 수정 | 수동 처방전(`kind:'MANUAL'`) | 자동 적용 안 함 → "수동 조치 필요" 목록으로 안내 |
| 간헐 실패(flaky)를 실패로 오판 | P1 Flaky Gate(확진/의심) | `SUSPECTED`(1/2)는 치료 대상에서 보류하거나 재확인 우선 |

**결론: 전체 치료 버튼이 "완치"라고 표시하는 것은 반드시 재진단으로 검증된 VERIFIED_RESULT뿐이다.** 이것이 "실제 치료" 보장의 근거다.

## 3. 동작 설계 (배치 오케스트레이터)

```
[💉 전체 치료] 클릭
  → baseline = 직전 전체 진단 결과 (lastRunResults)
  → 실패 진단 목록 수집 (status ∈ {ERROR, WARNING}, 단 SUSPECTED는 후순위/보류)
  → 각 진단 순차 처리:
       proposal = createRepairProposal(dir, diag, {strategy:'auto'})
       ├─ kind:MANUAL        → [수동 조치 필요]에 수집 (자동 적용 X)
       ├─ BLOCKED_WEAKENING  → [약화 차단]에 수집
       ├─ 처방 불가(키 없음+로컬 룰 없음) → [처방 불가]에 수집
       └─ 코드 수정 처방:
             result = applyRepairProposal(dir, proposal, {baselineResults: baseline})
             ├─ VERIFIED_RESULT → [✅ 실제 완치]  + baseline 갱신(치료 반영)
             └─ ROLLED_BACK     → [⚠️ 롤백(회귀/미완치)]
  → 최종 치료 리포트 반환:
       ✅ 실제 완치 N · 📋 수동 조치 M · ⚠️ 롤백 K · 🚫 약화 차단 · 🔑 처방 불가
```

- **baseline 갱신**: 한 건을 치료해 성공하면 그 결과가 다음 건의 회귀 판정 기준이 된다(누적 정확도 유지).
- **순서**: 로컬 룰·수동(무AI, 즉시)을 먼저, AI 처방을 나중에 → 429 쿼터 위험 최소화.

## 4. 구현 범위 (실행 스테이지에서 진행할 항목)

### B1. 백엔드 — 전체 치료 오케스트레이터
- `src/repairer.js`: `async function cureAll(projectDir, diagResults, { strategy='auto', chat, getByok } = {})` 신설. 위 3장 흐름을 서버에서 원자적으로 수행하고 분류별 요약 반환. 기존 `createRepairProposal`/`applyRepairProposal`을 재사용(신규 치료 로직 없음).
- `src/dashboard.js`: `POST /api/repair/cure-all` — body로 strategy 수신, `cureAll(currentProjectDir, lastRunResults, …)` 호출 후 요약 + 갱신된 전체 결과 반환. `lastRunResults`도 최신 재진단으로 교체.
- 치료 원장(P4)에는 각 건이 이미 기록됨(cureAll은 applyRepairProposal을 쓰므로 자동).

### B2. 프론트 — 💉 버튼 + 진행 + 리포트
- [App.jsx:758](dashboard-ui/src/App.jsx) 툴바의 진단 실행 버튼 옆에 `💉 전체 치료` 버튼(btn-primary, 톤 구분). **활성 조건: 실패 진단 ≥ 1**.
- 클릭 → (선택)확인 → `POST /api/repair/cure-all` → "치료 중… (진행 표시)" → **치료 리포트 모달**: 분류별 건수 + 각 항목(완치/롤백/수동/차단) 리스트. 수동 조치 건은 처방전 단계 노출.
- 완료 후 자동 재진단 반영 + 치료 원장 타임라인 갱신.

### B3. 테스트
- 혼합 시나리오 픽스처(로컬 완치 1 + 수동 1 + 회귀 유발 1 + 정상 1)에서 cureAll 요약이 `완치1/수동1/롤백1`로 정확히 분류되는지.
- **할루시네이션 미인정 검증**: 재진단이 OK가 아닌 처방은 절대 "완치"에 포함되지 않고 롤백되는지(가짜 OK 차단).
- 429/키 없음에서 로컬·수동은 처리되고 AI 필요 건만 "처방 불가"로 분류되는지.

### B4. 검증·문서
- 260714(수동 처방) + ESM 실험 프로젝트(로컬 완치) 실서버 E2E, 완료 보고서(Walkthrough).

## 5. 리스크와 완화
| 리스크 | 완화 |

## 6. 결정 게이트
- 권고: **Go** — 신규 치료 로직이 아니라 기존 안전 파이프라인의 배치 진입점이므로 위험이 낮고, "실제 치료" 보장은 이미 검증됨(VERIFIED_RESULT).
- 승인 시 순서: B1(백엔드 cureAll + API) → B3(테스트) → B2(💉 버튼·리포트 UI) → B4(E2E·문서). 각 단계 완료 시 6분류 보고, push는 매번 별도 승인.
