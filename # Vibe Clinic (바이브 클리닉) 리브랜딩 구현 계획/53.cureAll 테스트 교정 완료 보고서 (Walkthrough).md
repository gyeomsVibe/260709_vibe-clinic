## 2. 발견한 것 (근본 원인)

### 문제: ESM 진단 id 교체로 인한 `finalResults` 조회 실패

ESM `type:module` 프로젝트에서 `.clinic.js`가 로드 실패할 때, 진단 결과의 `id`는 **파일명** 기반(`esm`)으로 보고된다.
로컬 룰이 `.cjs`로 변환해 치료하면, 재진단 시 모듈이 실제로 로드되어 **모듈 내부 id**(`esm-diag`)로 교체된다.

```
치료 전 baseline id  : "esm"      (파일명 기반 — 로드 실패)
치료 후 finalResults id : "esm-diag"  (모듈 id — 로드 성공)
```

`cureAll`의 `cured` 배열에는 이미 이 교체를 반영한 `healedId: 'esm-diag'`가 기록되어 있었다.
그러나 테스트가 `c.diagId`(`'esm'`)로만 `finalResults`를 조회해 `undefined`를 반환 → `TypeError` 발생.

## 3. 수정한 것

### test/unit.test.js — L948~955

변경 전:
```
for (const c of report.cured) {
  const after = report.finalResults.find(r => r.id === c.diagId);
  assert.strictEqual(after.status, 'OK', ...);
}
```

변경 후:
```
// healedId(치료 후 실제 id)를 우선 조회하고, 없으면 diagId로 fallback한다.
for (const c of report.cured) {
  const after = report.finalResults.find(r => r.id === (c.healedId || c.diagId));
  assert.strictEqual(after?.status, 'OK', ...);
}
```

수정 근거: `cureAll`이 이미 `healedId`에 id 교체를 기록하고 있으므로, 테스트도 동일한 전략으로 일치시킨다.
할루시네이션 방지 검증(재진단 OK 필수) 로직 자체는 변경 없음.
